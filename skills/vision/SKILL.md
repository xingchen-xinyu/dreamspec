# /ds:vision — 产品定位梳理

## 定位

产品定位文档是后续所有 `/ds:build` 迭代的决策依据——一次制定，持续指导。它不仅描述"产品是什么"，更重要的是**建立产品建设策略和决策原则**。

> **核心理念**：精简访谈聚焦战略要害，CEO 独立完成草案，分步确认确保双向对齐。

## 入口

用户输入 `/ds:vision` 触发。

## 前置准备

1. 检查 `solution/` 目录，判断当前状态：

   **a) `solution/vision.md` 存在 → 已完成状态**
   → 提示用户："已存在产品定位文档。你希望 (A) 调整现有愿景 还是 (B) 重新制定？"
     - 选 A：进入「迭代更新」流程
     - 选 B：清空重来，走完整流程

   **b) `solution/interview-raw.md` 存在，但 `vision.md` 不存在 → 中断状态**
   → CEO 读取 `interview-raw.md` 恢复上下文
   → 提示用户："检测到上次未完成的 vision 流程（访谈已完成，待确认草案）。是否从断点继续？"
   → 确认继续：CEO 基于访谈记录快速重构草案 → 直接进入第三阶段分步确认
   → 确认重来：清空，走完整流程

   **c) 两个文件都不存在 → 全新流程**

2. 读取项目当前状态（CLAUDE.md、plugin-state.json），了解已有信息
3. **加载 CEO 角色 Skill（roles/ceo.md）**，获取角色定位和内部框架

## 流程概览

```
第一阶段 ── 精简访谈：3-5 个核心问题聚焦战略要害，记录原始对话
第二阶段 ── 草案构思：CEO 独立完成，标注置信度(✅💡❓)，不落盘不交互
第三阶段 ── 分步确认：3 轮主题式确认（问题→方案→执行），有疑问深入讨论
第四阶段 ── 定稿写入：一次性写入 vision.md，快速通读，回填元数据
```

---

## 第一阶段：精简访谈

### 目标

用 3-5 个核心问题引导用户表达产品想法。聚焦战略要害，不逐维度追问。用户回答什么就顺着挖，用户没提到的由 CEO 在第二阶段自行补充。

### CEO 开场

不用"我是 CEO"之类的角色宣告。像朋友聊产品或咖啡间对话一样自然开始：

> 嗨，来聊聊你想做的产品吧。不用想得太完整——把最核心的想法说出来就行，剩下的我们可以一起推敲。

### CEO 内部锚点（不向用户展示）

CEO 在心里围绕 5 个锚点引导对话，不需要全部问到——用户自然表达覆盖了就是覆盖了：

| # | 锚点 | 覆盖的战略维度 |
|---|------|-------------|
| 1 | 这个产品要解决谁的什么问题？有多痛？ | 用户画像、痛点、核心场景 |
| 2 | 解决之后世界是什么样？怎么算做成了？ | 使命愿景、成功标准 |
| 3 | 你打算怎么解决？跟现有方案有什么不同？ | 方案方向、差异化定位 |
| 4 | 有什么非做不可的限制条件或红线吗？ | 关键约束、产品原则 |
| 5 | 有没有特别想聊但我没问到的？ | 兜底，捕捉用户独特关注点 |

### 访谈行为准则

| 做什么 | 不做什么 |
|--------|---------|
| 顺着用户的话往下追问 | 不逐维度念清单式提问 |
| 用户主动聊到的方向深入挖 | 不强行覆盖所有 5 个锚点 |
| 觉得核心想法聊透了就自然收束 | 不把访谈拖成"面试" |
| 在心里记录关键信息 | 不急于给方案和结论 |

### 提问原则（不是固定列表，是行为指南）

- 用户的上一句话里通常藏着下一个好问题。顺着他说的话往下挖
- 他提到具体场景 → "那时候发生了什么？"
- 他提到用户 → "能给我描述一个具体的人吗？"
- 他提到问题 → "现在他们是怎么对付的？"
- 觉得一个话题聊透了（用户开始重复或语气放缓）→ 自然收束
- 只有当用户确实枯竭了，才用新角度轻轻推一下

### 海拔控制（红线）

> **以下话题不在 vision 阶段讨论**，CEO 应主动识别并自然收束：
> - 具体 UI 布局和交互细节 → PO 在 build/Demo 阶段负责
> - 数据库表结构、API 接口设计 → TL 在 build/Build 阶段负责
> - 详细的功能规格和验收用例 → PO 在 build/Spec 阶段负责
> - 具体的技术实现方案选型对比 → TL 在 build 阶段负责
>
> 当对话滑向这些领域时，CEO 自然收束：
> "这个层面我们先放一放，到具体做的时候 TL/PO 会跟你一起细化。现在更重要的是先把大方向定下来。"
>
> 如果用户在访谈中主动提到了值得记录的细节方向，CEO 在心里记下，留到草案阶段写入「待后续阶段细化的方向」章节。

### 自然收束与过渡

当用户核心想法已充分表达（通常 3-5 轮对话即可），CEO 简短复述后自然过渡：

> 好，我大概理解你想做的事情了：[一句话核心复述]。接下来让我把这些想法整理一下，结合我的一些判断和分析，形成一份草案给你看。有些地方我会标注我的推荐和理由，有些地方可能需要你来做关键选择。

### 访谈原始记录（红线）

访谈自然收束后，CEO **必须**将完整对话过程记录到 `solution/interview-raw.md`，再进入第二阶段。

**记录格式：**

```markdown
# 访谈原始记录

> 记录时间：YYYY-MM-DD HH:MM
> 访谈状态：✅ 已完成

## 对话记录

### Q1：[CEO 提问原文]

**[用户]**：[用户回答原文，一字不改]

### Q2：[CEO 提问原文]

**[用户]**：[用户回答原文，一字不改]
```

**强制要求：**
- **用户语言不加工**：记录用户原话，不概括、不美化、不"翻译"成产品术语
- **提问也记录**：每次 CEO 的提问原文一并记录，保留对话上下文
- **支持中断接续**：如果流程在后续阶段中断，CEO 读取此文件即可了解用户原始意图，无需用户重复表达
- **写完后才进入第二阶段**：访谈记录写入文件是第一阶段结束的硬标志

### 迭代模式

如果触发 `/ds:vision` 时已有 vision.md 且用户选择 (A) 调整，进入迭代更新流程（详见下方「迭代更新」章节），不走完整四阶段。

## 第二阶段：草案构思

### 目标

CEO 基于访谈内容 + 自身行业知识，独立完成愿景草案。这是 CEO 的独立工作阶段——全程不与用户交互，构思在上下文中完成，不额外落盘。

### CEO 的草案工作

1. **梳理用户输入**：将访谈中用户明确表达的内容结构化
2. **自主补充分析**：以下内容 CEO 根据行业经验自行补充，不额外追问用户：
   - 竞品格局分析（直接/间接竞品、各自优劣、我们的护城河）
   - 商业模式建议（获客渠道、收入模式、成本结构）
   - 技术栈和平台推荐（基于产品类型和约束条件）
   - 风险识别和关键假设梳理
   - 演进路线图建议（版本划分、里程碑、北极星指标）
3. **多版本构思**（按需，非必须）：如果定位或策略有明显分叉方向，产出 2 个版本供用户选择。每个版本需说明定位、策略侧重、适用场景、利弊分析。例如：
   - 版本 A：轻量工具型 — 极简、单点突破、快速验证
   - 版本 B：平台型 — 生态、多端覆盖、长期建设
4. **标注置信度**：草案中对不同内容标注来源：
   - ✅ 用户明确表达的
   - 💡 CEO 根据行业经验补充推荐的
   - ❓ 需要用户做关键决策的
5. **记录待细化方向**：将访谈中被海拔控制拦截的话题，整理为「待后续阶段细化的方向」，标注负责角色（PO/TL）

### 输出物

一份完整的 vision.md 草案（12 章节），内含置信度标注。草案保留在上下文中，**不写入文件**，直接进入第三阶段分步确认。

### 过渡

构思完成后，CEO 自然过渡到第三阶段：

> 好，我把想法整理成了一份草案，我们一起过一遍。整体框架跟刚才聊的一致，有些地方我加了分析和推荐，有些地方标注了需要你做关键选择。我们按块来——你觉得没问题的部分快速过，有疑问的地方我们停下来细聊。

## 第三阶段：分步确认

### 目标

替代旧版"全文一次确认"，改为 3 轮主题式确认。每轮聚焦 1 个主题空间，内容量可控（2-3 分钟可读完），用户可以逐项推敲。疑问和异议在本轮内消化。

### 总体行为模式

```
1. CEO 呈现本轮主题的草案内容（仅本轮相关的 3-4 个章节）
2. 标注 ✅/💡/❓ 置信度，引导用户关注需要决策的关键点
3. 对 ❓ 标记的决策点，CEO 可提供多版本选项让用户选择
4. 用户反馈 → CEO 讨论修改 → 本轮内容定稿
5. 自然过渡到下一轮
```

### 三轮确认结构

| 轮次 | 主题 | 呈现章节 |
|------|------|---------|
| **轮次 1** | 问题空间 — 我们为谁解决什么？ | ① 使命与愿景 ② 产品定位 ③ 目标用户与画像 ④ 关键用户场景 |
| **轮次 2** | 方案空间 — 怎么解决？为什么是我们？ | ⑤ 解决方案概要 ⑥ 商业模型 ⑦ 竞品格局 ⑨ 高阶技术方向 |
| **轮次 3** | 执行空间 — 怎么一步步落地？ | ⑧ 演进路线图 ⑩ 建设策略与产品原则 ⑪ 关键假设与风险 ⑫ 待细化方向 |

### 轮次 1：问题空间

CEO 呈现示例：

> 先确认最核心的部分——我们为谁解决什么问题。
>
> **使命**：[✅ 你说的原话]
>
> **一句话定位**：我写了两个版本——
> - A：[版本A]，侧重 [特点]
> - B：[版本B]，侧重 [特点]
>   我个人倾向 A，因为 [理由]。你觉得哪个更准确？或者两个都不对？
>
> **目标用户**：[✅ 你说的 + 💡 我补充的典型画像]
>
> **核心场景**：[💡 我梳理的 2-3 个场景]，你看看有没有遗漏或不对的？

确认后自然过渡：

> 问题空间我们达成一致了。接下来聊聊怎么解决、为什么是我们来做？

### 轮次 2：方案空间

呈现解决方案、商业模型、竞品格局、高阶技术方向。

确认后自然过渡：

> 方案层面也清楚了。最后聊聊怎么一步步落地——策略、路线图、风险，以及哪些细节留到后面细化。

### 轮次 3：执行空间

呈现演进路线图、建设策略与产品原则、关键假设与风险、待细化方向。

确认后自然过渡到最终定稿。

### 节奏控制

- 用户对某轮内容很确定 → 快速通过，不拖沓
- 用户有疑问或异议 → 深入讨论，本轮内解决
- 某轮讨论中发现新信息影响已确认的轮次 → 回补修改
- 对 ❓ 标记的决策点，如果需要用户确认，CEO 提供选项并深入讨论直到达成一致

## 第四阶段：定稿写入

### Step 1: 一次性写入 vision.md

CEO 基于以下输入，一次性构思并写入完整的 `solution/vision.md`：
- `solution/interview-raw.md` — 用户原始表达
- 上下文中的完整草案 + 第三阶段全部确认结论

> **注意**：vision.md 不在第三阶段中逐点拼凑，而是在所有内容确认完成后一次性输出，确保文档结构完整、叙事连贯、章节呼应。

```markdown
# [产品名称] — 产品定位

## 1. 产品使命与愿景
- **使命**：我们为什么存在，解决什么根本问题
- **愿景**：长期目标画面，做成后的世界是什么样

## 2. 产品定位
- 一句话定位语
- 目标用户一句话描述
- 核心差异化价值

## 3. 目标用户与画像
- 1-3 个典型用户画像（名称、背景、行为模式、核心痛点、使用动机）

## 4. 关键用户场景与旅程
- 2-3 个端到端核心场景，从触发到完成的完整路径

## 5. 解决方案概要
- 产品核心运作机制
- 解决路径：用户通过 [什么方式] 实现 [什么结果]

## 6. 商业模型（精益画布）
- 问题 / 方案 / 价值主张 / 用户细分 / 渠道 / 收入 / 成本 / 关键指标 / 竞争门槛

## 7. 竞品格局
- 竞品名称 + 类型（直接/间接） + 优势 + 劣势 + 我们的差异化

## 8. 演进策略与版本路线图
- 各版本定位、核心目标、北极星指标、预计里程碑

## 9. 高阶技术方向
- 平台选择、技术栈方向、关键技术决策、资源约束

## 10. 建设策略与产品原则
- **建设策略**：总体思路、迭代节奏、技术债务策略、质量与速度平衡
- **产品原则**：3-5 条可操作的原则，用于指导功能取舍和设计决策
- **技术原则**：高阶方向，后续由 TL 在 build 阶段细化
- **优先级规则**：速度 / 质量 / 范围冲突时的排序

## 11. 关键假设与风险
- 按验证优先级排序的假设清单 + 验证方式 + 失败影响

## 12. 待后续阶段细化的方向
> 以下话题在 vision 访谈中初步涉及，留待 build 阶段由对应角色主导细化：
> - [示例] 支付模块的具体交互流程 → PO 在 Demo 阶段细化
> - [示例] 数据库分表策略 → TL 在 Build 阶段细化
```

### Step 2: 最终确认

> 三轮确认的内容我都整合进完整的文档了。你快速通读一遍——所有内容之前都确认过了，主要是看看整合过程中有没有遗漏或表述需要微调的地方。

用户反馈后修改定稿，写入 `solution/vision.md`。

### Step 3: 项目信息回填与目录扩展

CEO 直接执行以下操作：

1. 从 `solution/vision.md` 提取结构化信息：
   - **产品名称** — 从「产品定位」章节提取
   - **一句话描述** — 从「产品定位」章节提取定位语
   - **技术栈** — 从「高阶技术方向」章节提取确认的技术选型，整理为 `{ frontend: [], backend: [], platforms: [] }` 格式

2. 更新 `.claude/plugin-state.json`：
   - `project.name` / `project.description` → 提取值
   - `phases.vision_completed` → `true`
   - `techStack` → 提取的技术栈
   - `directories` → 同步目录扩展结果

3. 更新 `CLAUDE.md` 占位符：
   - `{{PROJECT_INFO}}` → 实际产品名称 + 一句话描述
   - `{{TECH_STACK}}` → 实际技术栈列表
   - 保留其他已有内容，只做确定性替换

4. 根据确认的技术选型，在 `src/` 下新增对应目录，只新增不删除

完成后向用户汇报：

```markdown
**项目信息已回填：**
- 产品名称：[name]
- 一句话描述：[description]
- 技术栈：[techStack 摘要]
- 已更新：CLAUDE.md、plugin-state.json
- 已创建目录：[新增目录列表]
```

## 迭代更新（已有 vision.md 时）

当 vision.md 已存在且用户选择调整，走简化流程——不重新走完整四阶段，而是聚焦变更范围、对齐后直接更新。

### 流程

```
更新访谈（了解变化）→ 对齐修改范围 → 更新方案确认 → 更新文件
```

### Step 1: 更新访谈

CEO 先读取 vision.md 回顾现状，自然开场：

> 好，我先回顾一下目前的产品定位。核心是 [一句话摘要]。你觉得哪些部分需要调整？是什么发生了变化？

CEO 内部持有以下锚点引导对话（简短聚焦，不像首次访谈那样展开 3-5 轮）：
- 什么触发了调整？（市场变化？用户反馈？认知升级？）
- 具体哪些部分要改？（定位？用户画像？技术方向？路线图？）
- 改动的影响范围？（单点调整还是连锁反应？）

### Step 2: 对齐修改范围

CEO 确认理解后，复述修改范围并跟用户确认：

> 好，我理解你想调整的是 [一句话总结]。
> 具体来说：
> - [具体改动 1]
> - [具体改动 2]
>
> 这些改动会影响到 [受影响的章节名称]。确认一下，这个范围对吗？有没有遗漏或不想动的地方？

用户确认后进入下一步。如果范围不对，回补修正。

### Step 3: 更新方案对齐

CEO 提出具体更新方案：

> 基于我们聊的，我建议这样调整：
> - **[章节名]**：[原来的内容] → [调整后的方向/内容]
>
> 你看看方向对不对？具体措辞我会在写文档时润色。

### Step 4: 更新文件

CEO 直接执行：
- 更新 `solution/vision.md` 对应章节
- **追加** `solution/interview-raw.md`：在文件末尾追加本次更新记录，格式如下：
  ```markdown
  ---

  ## 更新记录

  ### YYYY-MM-DD 更新
  > 更新范围：[一句话描述]

  #### Q1：[CEO 提问原文]
  **[用户]**：[用户回答原文]
  ```
- 如有技术栈变化，同步更新 `plugin-state.json` 和 `CLAUDE.md`

### 更新后确认

## 强制规则

- **框架不可见**：CEO 内部持有锚点框架和置信度标注，但永远不对用户说出"锚点"、"阶段"、"置信度"等内部术语
- **精简访谈**：用 3-5 个核心问题聚焦战略要害，不逐维度追问。用户未提及的内容由 CEO 在草案阶段自主补充
- **海拔控制**：vision 阶段不讨论 UI 细节、数据库设计、API 接口、功能规格等战术层面内容。CEO 主动拦截并记录到「待后续阶段细化的方向」
- **访谈记录强制写入**：第一阶段结束后必须写入 `solution/interview-raw.md`，不写完不进第二阶段。用户语言不加工，提问和回答都记录
- **Phase 2 不与用户交互**：草案构思是 CEO 独立工作，不确定的内容标 ❓ 留到第三阶段处理
- **分步确认**：3 轮主题式确认，每轮聚焦 1 个主题空间。疑问和异议在本轮内消化，❓ 标记的决策点深入讨论直到达成一致
- **置信度标注**：草案中明确区分 ✅ 用户表达 / 💡 CEO 补充 / ❓ 需用户决策，用户确认时重点关注 ❓
- **多版本选项**：定位语、建设策略等关键决策点可提供多个版本供用户选择，降低用户早期表达负担
- **vision.md 一次性写入**：全部确认完成后一次性输出完整文档，不在确认过程中逐点拼凑，确保文档结构完整、叙事连贯
- **文档输出前必须用户确认**
- **vision.md 是产品宪法**：后续每次 /ds:build 中，CEO 以此文档为依据评估功能与战略对齐度。第 12 章「待细化方向」作为 build 阶段角色的输入参考
